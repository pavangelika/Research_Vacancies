# Этап 3. Реализация MVP-набора автотестов

Дата: 16.03.2026  
Статус: `в работе`

## 1) Цель этапа

Начать реализацию MVP-набора автотестов из baseline Этапов 1-2 без переписывания уже существующих тестов.

## 2) Что уже есть на старте этапа

1. В `tests/api` реализованы внешние API-проверки для `GET /vacancies` и `GET /vacancies/{id}`.
2. В `docs/report-testing-stage2.md` зафиксирован переход к реализации MVP-набора.
3. В проекте уже существует локальный server layer: `scripts/report_server.py` с ручками `/api/vacancies/*`.

## 3) Решение для текущего шага

В рамках перехода к Этапу 3:

1. Существующие тесты в `tests/api` сохраняются без переписывания.
2. Реализация расширяется в незакрытую P0-область: локальный API `report_server`.
3. Добавляется изолированный тестовый набор для `/api/vacancies/responses`, `/api/vacancies/details`, `/api/vacancies/send-resume`.
4. Новые тесты строятся без обращения к реальной БД: DB-функции мокируются на уровне `scripts.report_server`.

## 4) Покрытие, добавленное на Этапе 3

Новый набор покрывает:

1. `GET /api/vacancies/responses` -> успешный ответ `200`.
2. `GET /api/vacancies/details` -> `400`, если не передан `vacancy_id`.
3. `GET /api/vacancies/details` -> `404`, если запись не найдена.
4. `POST /api/vacancies/send-resume` -> `400` при невалидном JSON.
5. `POST /api/vacancies/send-resume` -> `404` для неизвестной вакансии.
6. `POST /api/vacancies/send-resume` -> `200` для успешного обновления.
7. `POST /api/vacancies/details` -> проброс сервисного результата сохранения.

## 5) Что осталось из P0/P1 после этого шага

Еще не реализованы:

1. DB-инварианты после API/UI-операций (`RQ-21`).
2. Проверки расчетов и эталонных агрегатов (`RQ-01`, `RQ-02`, `RQ-03`, `RQ-10`, `RQ-15`, `RQ-17`, `RQ-18`).
3. Проверки артефактов отчета и fallback-поведения (`RQ-04`, `RQ-07`, `RQ-08`).
4. UI smoke/e2e для вкладок и форм (`RQ-05`, `RQ-06`, `RQ-11`-`RQ-14`).

## 6) Предложения по улучшению уже существующих тестов `tests/api`

Текущие тесты не переписывались, но для следующего шага рекомендуется:

1. Убрать дублирование параметров запроса через общие фикстуры/фабрики.
2. Снизить зависимость от живого внешнего API в регрессионных сценариях с большим числом страниц.
3. Разделить smoke и тяжелые regression-проверки по отдельным наборам запуска.
4. Уточнить JSON Schema по реальным nullable-полям HH API, чтобы снизить ложные падения.
5. Привести проверки дат к явной timezone-aware логике.
6. Вынести повторяющийся Allure attach в единые helper-функции.

## 7) Критерий завершения текущего шага Этапа 3

1. Новый набор локальных API-тестов собирается и запускается изолированно.
2. Существующие тесты не переписаны.
3. Улучшения для существующих тестов зафиксированы как рекомендации, а не как переработка текущего набора.
