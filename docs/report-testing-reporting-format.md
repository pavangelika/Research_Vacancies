# Формат отчетности по прогонам автотестов (baseline)

Дата: 10.03.2026  
Статус: черновик для согласования

## 1) Каналы отчетности

1. Allure Report - детальные результаты тестов.
2. CI Summary - короткий итог в пайплайне.

## 2) Обязательные поля CI Summary

1. Commit/branch.
2. Время запуска и длительность.
3. Количество тестов: total/passed/failed/skipped.
4. Отдельно по наборам: `api`, `ui`, `db`, `data/logic`, `integration`.
5. Ссылка на Allure отчет.
6. Статус quality gate (`pass/fail`).

## 3) Quality Gate (первичный)

1. PR pipeline:
1. Все smoke и P0-проверки должны пройти.
2. Допустимо: не более 0 flaky-fail в обязательном наборе.
2. Nightly pipeline:
1. Полная регрессия без критических падений.
2. При падении P0 - автоматический алерт владельцам.

## 4) Классификация дефектов для отчетов

1. `critical`: искажение бизнес-метрик/итогов.
2. `high`: некорректная логика секций отчета, влияющая на интерпретацию.
3. `medium`: частичная деградация отображения/фильтров.
4. `low`: некритичные форматные/текстовые дефекты.

## 5) SLA эскалации

1. `critical` - реакция в течение рабочего дня.
2. `high` - до следующего релизного цикла.
3. `medium/low` - по приоритету бэклога.

## 6) Политика вложений (для всех автотестов)

1. Для каждого шага каждого автотеста формируется вложение в отчет (`API`, `UI`, `DB`, `data/logic`, `integration`).
2. Цель: полная трассируемость и возможность ручной проверки работоспособности шага в первом и последующих прогонах.
3. Минимальный состав вложения по шагу:
1. шаг/действие;
2. входные данные (параметры, payload, sql, ui-действие);
3. фактический результат (response/body, db-result, ui-state);
4. assert/ожидание;
5. статус шага.
4. Для `API` обязательно прикладывать метод, endpoint, params/body, status code, response body.
5. Для `UI` обязательно прикладывать действие + скриншот состояния шага (или эквивалентный артефакт).
6. Для `DB` обязательно прикладывать sql/операцию и фактический набор ключевых полей результата.
7. Перед сохранением вложений обязательна маскировка секретов и чувствительных данных.

## 7) Политика хранения и очистки артефактов

1. Автоочистка выполняется каждые 3 дня.
2. Артефакты `passed/skipped` удаляются при очистке.
3. Артефакты `failed` сохраняются (исключение из очистки) для ретроспективного анализа.
4. Период хранения `failed` устанавливается в CI (рекомендуемо 14-30 дней).
