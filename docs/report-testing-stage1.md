## 1) Цель этапа

Зафиксировать единый baseline для дальнейшей автоматизации тестирования системы в трех контурах: `API`, `UI`, `DB`.  
На Этапе 1 фиксируем границы, риски, критерии и стек. Реализация автотестов выполняется на следующих этапах.

## 2) Объект тестирования (единый контур API/UI/DB)

Поток системы:

1. Внешний API-источник: `https://api.hh.ru/vacancies` и `https://api.hh.ru/vacancies/{id}`.
2. Локальный API приложения (report server):  
`POST /api/vacancies/send-resume`, `POST /api/vacancies/details`,  
`GET /api/vacancies/responses`, `GET /api/vacancies/details`.
3. DB-слой: PostgreSQL таблицы `get_vacancies`, `employers` (+ поля отклика/интервью в `get_vacancies`).
4. Слой агрегации/расчетов: SQL + Python в [`scripts/generate_report.py`](/C:/Users/pavan/PycharmProjects/Research_Vacancies/scripts/generate_report.py).
5. UI-слой: Jinja2-шаблон [`report_template.html`](/C:/Users/pavan/PycharmProjects/Research_Vacancies/reports/templates/report_template.html) + `reports/static/*.js` + `styles.css`.
6. Результат: `reports/report.html` и API/DB-данные, отображаемые во вкладках отчета.

Критичные зоны отчета:

1. Корректность агрегатов по месяцам/ролям/опыту.
2. Корректность логики активных и архивных вакансий.
3. Зарплатные расчеты (avg/median/mode/min/max, распределение по исходным валютам).
4. Аналитика работодателей (факторы и группировки).
5. Корректная передача данных в HTML-атрибуты (`data-*`) и отображение вкладок/таблиц/графиков.
6. Сохранение данных из формы вкладки "Мои вакансии" в БД (поля интервью/результатов/комментариев).
7. Расчеты эффективности откликов.
8. Фильтрация вакансий по навыкам, валюте, стране и атрибутам работодателя (включая ИТ-аккредитацию).
9. Сортировка, а также логика условий "И"/"ИЛИ" в поиске по навыкам.
10. Расчет зарплат в разных валютах по исходной валюте.
11. Расчет топ-навыков по ролям/опыту/периодам.
12. Расчеты трендов рынка (динамика публикаций, архивирования, средней длительности жизни вакансии).
13. Контрактность и корректные коды/ошибки API в штатных и невалидных сценариях.

## 3) Scope / Out of Scope для первой итерации автоматизации

`In scope`:

1. Валидация структуры и качества входных данных для отчета.
2. Проверка бизнес-расчетов на эталонном наборе данных.
3. Проверка целостности сгенерированного `report.html` (ключевые блоки/секции/плейсхолдеры).
4. Проверка экспортируемых артефактов генератора (`report.html`, CSS, JS, plotly bundle).
5. Проверка записи/обновления данных в БД из формы "Мои отклики".
6. Проверка фильтрации/сортировки и логики "И"/"ИЛИ" в поиске по навыкам.
7. Проверка расчетов эффективности откликов.
8. Проверка расчетов зарплаты по исходным валютам.
9. Проверка расчетов топ-навыков.
10. Проверка расчетов трендов рынка.
11. API-тестирование: smoke, contract, негативные сценарии, проверка кодов ошибок и обязательных полей.
12. DB-тестирование: целостность записи/чтения, обновления и сохранность бизнес-полей.
13. UI-тестирование: smoke/e2e сценарии по вкладкам, фильтрам, формам и отображению результатов.
14. Единая политика вложений для всех автотестов: вложение на каждый шаг с последующей регулярной очисткой артефактов.

`Out of scope`:

1. Визуальный pixel-perfect контроль верстки.
2. Кросс-браузерная матрица (полный UI-регресс).
3. Нагрузочные прогоны и профилирование SQL/JS.
4. Изменение текущей бизнес-логики генерации отчета.

## 4) Риск-матрица (приоритет на этапах 2-5)

P0 (критично):

1. Искажение итоговых метрик (ошибка в SQL/Python-агрегации).
2. Потеря строк или некорректная фильтрация при сборке отчетных данных.
3. Неверные значения в зарплатной аналитике.
4. Потеря/искажение данных при записи формы "Мои отклики" в БД.
5. Ошибки расчета эффективности откликов.
6. Ошибки API-контракта/кодов ответа для критичных ручек.

P1 (высокий):

1. Неконсистентность между данными и отображаемыми секциями отчета.
2. Поломка структуры `data-*` для JS-рендеринга.
3. Ошибки fallback-поведения (например, отсутствие данных по роли/месяцу).
4. Некорректная фильтрация по навыкам/валюте/стране/атрибутам работодателя.
5. Ошибки сортировки и логики "И"/"ИЛИ" при поиске по навыкам.
6. Ошибки расчетов топ-навыков и трендов рынка.
7. Неконсистентность между UI-формой, API-ответом и фактической записью в БД.

P2 (средний):

1. Некритичные форматные отклонения.
2. Неполные тексты/лейблы в отдельных UI-блоках.

## 5) Критерии входа/выхода

### Entry Criteria

1. Актуальная версия генератора отчета в `main`/рабочей ветке.
2. Доступен тестовый PostgreSQL-контур и фиксированный набор тестовых данных.
3. Зафиксированы формулы и правила агрегации для ключевых метрик.
4. Определены обязательные секции HTML-отчета.
5. Для будущих UI-проверок подготовлен запуск отчета через локальный HTTP-сервер (не `file://`) для стабильной работы Playwright.
6. Определен тестовый набор API-ручек (внешних и локальных) и ожидаемые коды/схемы ответа.
7. Определены DB-инварианты для полей отклика/интервью и отчетных агрегатов.

### Exit Criteria (для завершения Этапа 1)

1. Утвержден текущий документ как baseline стратегии подготовки.
2. Утверждены `in scope`/`out of scope`.
3. Утверждены P0/P1 риски и критерии приемки.
4. Зафиксирован целевой стек автоматизации и формат отчетности.
5. Зафиксирован факт согласования (дата, роли, решение) в разделе "Лист согласования".
6. Утверждена единая политика вложений и хранения артефактов для всех автотестов.

## 6) Целевой стек (зафиксирован на этапе подготовки)

1. Общий раннер: `pytest`.
2. API: `requests`/`pytest` + `jsonschema` (contract/smoke/negative).
3. DB: `psycopg2` + SQL-проверки целостности и бизнес-инвариантов.
4. Расчеты и табличные проверки: `pandas` (+ сравнение с эталонными таблицами).
5. UI: `Playwright` (smoke/e2e для вкладок, фильтров, формы "Мои вакансии").
6. HTML-валидация артефактов: `BeautifulSoup/lxml`.
7. Отчетность запусков: `allure`.
8. CI-исполнение: GitHub Actions (`smoke` на PR, полная регрессия nightly) - подключение на следующих этапах.

## 7) Роли и ответственность

1. QA (владелец автотестов): проектирование, реализация, поддержка тестов.
2. Аналитик/владелец отчета: подтверждение формул, бизнес-правил и эталонных результатов.
3. Разработчик: поддержка тестируемости генератора, фиксы дефектов.
4. Ответственный за релиз: решение о приемке на основании отчетов тестов.
