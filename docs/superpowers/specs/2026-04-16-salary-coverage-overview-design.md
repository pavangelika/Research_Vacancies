# Дизайн: карточка "Покрытие зарплат" в totals overview

## Цель

Добавить в overview-дэшборд новую карточку, которая показывает:

- сколько вакансий без зарплаты
- сколько вакансий с зарплатой
- из вакансий с зарплатой разбивку по валютам:
  - `RUR`
  - `USD`
  - `EUR`
  - `Другая`

Карточка должна быстро отвечать на два вопроса:

1. Насколько текущий срез вакансий вообще покрыт зарплатой.
2. В каких валютах представлены вакансии с зарплатой.

## Область изменений

Изменения ограничены frontend-логикой overview-блока в `totals`.

Не входят в задачу:

- backend-изменения
- новые API
- изменение структуры исходных вакансий
- изменение существующей карточки `Зарплаты`
- изменение логики фильтров

## Источник данных

Карточка строится на том же отфильтрованном массиве `vacancies`, который уже используется в `renderGlobalTotalsFiltered(...)` для totals-виджетов.

Это значит, что карточка автоматически уважает текущие фильтры:

- роли
- период
- опыт
- статус
- валюта
- страна
- остальные глобальные фильтры, которые уже применяются к totals

Дополнительный источник данных не нужен.

## Модель данных

Нужен отдельный агрегатор, который принимает массив вакансий и возвращает:

- `total`
- `withSalary`
- `withoutSalary`
- `currencies.RUR`
- `currencies.USD`
- `currencies.EUR`
- `currencies.other`

### Правила расчёта

Вакансия считается `с зарплатой`, если в ней есть вычисляемое зарплатное значение по уже принятой в проекте логике зарплаты.

Вакансия считается `без зарплаты`, если зарплатное значение не определяется.

Разбивка по валютам строится только для вакансий `с зарплатой`.

Нормализация валют:

- `RUR` -> `RUR`
- `USD` -> `USD`
- `EUR` -> `EUR`
- всё остальное непустое -> `other`
- пустая/неопределённая валюта при наличии зарплаты -> `other`

## UI-структура

Новая карточка называется `Покрытие зарплат`.

Внутри карточки два уровня:

### Верхний уровень

Два крупных блока:

- `С з/п`
- `Без з/п`

Для каждого блока показываются:

- абсолютное количество
- доля в процентах от общего числа вакансий текущего среза

### Нижний уровень

Четыре компактных блока:

- `RUR`
- `USD`
- `EUR`
- `Другая`

Для каждого блока показываются:

- абсолютное количество
- доля в процентах от количества вакансий `с з/п`

## Визуальный стиль

Карточка должна быть HTML-based, без Plotly.

Причины:

- карточка маленькая и summary-ориентированная
- значения считываются быстрее в компактной HTML-вёрстке
- не создаём лишнюю визуальную конкуренцию с `Сгорание вакансий` и `Зарплаты`

Цветовая схема:

- `С з/п` -> основной сине-голубой семейства totals
- `Без з/п` -> нейтральный серо-холодный
- валюты -> одна согласованная шкала оттенков, без конфликта с активными/архивными цветами

## Расположение

Карточка добавляется в `dashboard-overview` рядом с существующими карточками overview.

Предпочтительно располагать её рядом с карточкой `Зарплаты`, потому что это одна смысловая группа.

## Пустые состояния

Если вакансий нет:

- верхние значения показываются как `0`
- проценты показываются как `0%`
- нижняя валютная разбивка тоже показывается как нули

Карточка не скрывается.

## Поведение и совместимость

Карточка не добавляет интерактивности первого этапа:

- без drilldown
- без кликов по сегментам
- без дополнительных фильтров

Это intentionally scope-limited summary-widget.

## Тестирование

Нужно проверить:

1. Карточка появляется в overview без поломки layout.
2. Сумма `withSalary + withoutSalary` равна `total`.
3. Сумма валют `RUR + USD + EUR + other` равна `withSalary`.
4. Карточка корректно меняется при переключении фильтров totals.
5. При пустом срезе отображаются нули, а не `NaN`/`undefined`.

## Риски

Основной риск — разойтись с уже существующей логикой определения наличия зарплаты.

Чтобы этого избежать, новый агрегатор должен переиспользовать текущие salary-helper conventions проекта, а не вводить собственную альтернативную трактовку зарплаты.

## Рекомендованный подход

Реализовать отдельный агрегатор для salary coverage и отдельный HTML-render helper для новой карточки.

Это лучший вариант, потому что:

- не перегружает `buildSalaryChartHtml(...)`
- не смешивает salary coverage с диапазонами зарплат
- сохраняет понятные границы между расчётом и отображением
